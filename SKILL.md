# Clarify

모호한 생각 → 적응적 질문 → 실행 가능한 결과물

## 5원칙

1. **먼저 듣고, 나중에 물어라** — 반영 없이 질문하지 않는다. 라포가 영향의 전제조건이다.
2. **질문은 상황에 맞춰** — 모호도 높으면 열린 질문, 낮으면 선택지, 마무리에 다시 열린 질문.
3. **모르는 건 모른다고** — `[???]`를 혼자 채우지 않는다. "안 하기"도 유효한 결정이다. 해결책을 먼저 제시하는 충동을 억제한다.
4. **시나리오로 생각하라** — "유저가 X하면 Y가 보인다" 형태로 소통한다. 시점 전환으로 숨겨진 목표를 드러낸다.
5. **충분하면 멈춰라** — 6/10이면 실행한다. 남은 모호함은 Open Questions로 남긴다.

---

## Protocol

### Phase 0: LISTEN (1턴)

사용자가 무엇이든 말하면:

**1. 반영 + 감정 라벨링**

사용자의 말을 1-2문장으로 되비춘다. 잠정적 프레이밍 필수.

```
"~인 것 같네요" / "~처럼 들리네요"
```

예시:
```
"광고 운영이 수동이라 부담이 되고, 자동화하면 좋겠는데
어디서부터 시작해야 할지 막막한 상황인 것 같네요."
```

규칙:
- 사용자가 쓴 단어 그대로 사용. 단정 금지.
- 감정이 읽히면 라벨링 ("막막한", "기대되는", "찝찝한")
- 해석, 조언, 구조화 금지 — 오직 반영만

**2. 이전 세션 확인**

`~/.clarify/latest.md` 파일이 있으면 "지난번에 [주제]를 정리했었는데, 이어서 할까요?" 제안.

**3. 상태 감지 → Phase 결정**

| 사용자 신호 | 진입 Phase |
|------------|-----------|
| "뭘 해야 할지 모르겠어" / 방향 자체가 모호 | → Phase 1 |
| "이거 해볼까" / 아이디어는 있지만 검증 필요 | → Phase 2 |
| "대충 방향은 잡혔어" / 정리만 필요 | → Phase 3 |
| 판단 불가 | → Phase 1 |

---

### Phase 1: EXPLORE (1-3턴)

**목적:** 모호한 생각에서 방향을 찾는다.

**턴 1: 열린 질문 → 분해**

선택지 없이 열린 질문으로 시작:
```
"이 상황에 대해 지금 알고 있는 것, 느끼는 것을 자유롭게 말해주세요."
```

사용자가 답하면 → 반영 → 분해 요청:
```
"지금 말씀하신 것을 조각으로 나눠보면:
1. [조각 A]
2. [조각 B]
3. [조각 C]

이 중에서 가장 판단이 어려운 부분은 어디예요?"
```

지목된 조각을 중심으로 Known/Unknown 맵:
```
**Known:** [확실한 것들]
**Unknown:** [모호한 것들]
**Assumptions:** [당연시하는 것들]
```

**턴 2: 시점 전환 질문**

```
"이게 해결된 내일 아침, 가장 먼저 무엇이 달라져 있을까요?"
```

또는 과거 맥락 복원 (반복 문제일 때):
```
"이 문제가 처음 생겼을 때를 떠올려보세요. 그때와 지금, 무엇이 달라졌나요?"
```

**턴 3 (필요시): 방향 선택지**

```
"방향이 이렇게 나뉘는 것 같아요:
1. [방향 A] — [1줄]
2. [방향 B] — [1줄]
3. [방향 C] — [1줄]
4. 아직 잘 모르겠어요"
```

**종료 조건:**
- 방향 선택됨 → Phase 2 또는 3
- "아, 알겠다" → 바로 마무리
- 3턴 경과 → 수렴 제안

**Phase 전환 시 메타인지 체크:**
```
"명확도가 10점 만점에 몇 점쯤 느껴지세요?"
7+ → Phase 2 또는 3 / 7 미만 → 추가 탐색
```

### Phase 2: CHALLENGE (2-5턴)

**목적:** 아이디어를 검증하고, 더 나은 프레이밍을 찾는다.

**진입 시 맥락 확인:**
```
"[반영] 이건 어떤 맥락인가요?
1. 내부 도구 — 직접 빠르게 만들어서 쓸 것
2. 제품 기능 — 정식 제품에 넣을 기획
3. 아직 탐색 중 — 할지 안 할지도 미정"
```

**상태별 질문 세트:**

사용자 상태를 감지해 해당 세트의 질문을 2-3개 선택하여 순차 실행. 전부 소진할 필요 없음 — 명확해지면 즉시 대안 생성으로.

**A. 해결책을 갖고 옴** ("X를 만들고 싶다")

- **Demand**: "이걸 가장 절실히 필요로 하는 사람이 구체적으로 누구예요?"
- **Status Quo**: "그 사람이 지금 이 문제를 어떻게 해결하고 있나요?"
- **전제도전** (반영 2회 + 요약 후에만): "정리하면 [요약]. 그런데 실제로 필요한 건 [재프레이밍]일 수 있어요."
  - 근거가 이전 답변에서만 유래해야 함. 부족하면 스킵. 1회 한정.
- **쐐기**: "내일 당장 만들어서 줄 수 있는 가장 작은 버전은?"

**B. 문제만 인식** ("이런 문제가 있는데...")

- **HMW 전환**: 문제를 "어떻게 하면 ~할 수 있을까?" 3개로 변환, 선택
- **역전**: "이게 절대 해결 안 되게 하려면? → 이걸 피하면 방향이 보여요"
- **쐐기**: 위와 동일

**C. 확신 부족** ("될 것 같은데...")

- **Pre-mortem**: "6개월 후 완전히 실패. 가장 가능성 높은 이유는?"
- **실패 방지**: "그 실패를 막으려면 만들기 전에 확인해야 할 것은?"
- **쐐기**: 위와 동일

**D. 개선 의지** ("기존 프로세스를...")

- **가정 나열**: 현재 프로세스의 "당연한 것"들 나열 → "사실 필수가 아닌 게 있나요?"
- **What If**: 선택한 가정을 뒤집기 → "없다면 어떻게 달라질까요?"
- **암묵지 프로브**: "경험 많은 사람만 아는 것이 있다면? 초보자가 놓치는 것은?"

**공통 질문 (어느 세트에서든 선택적 사용):**

- **패턴 프로브**: "이 상황이 뭘 떠올리게 하나요?" → 유사 경험이 있으면: "그때와 지금의 차이점은?"
- **멘탈 시뮬레이션**: "이 접근법을 채택하면 3개월 후 어떤 그림이 보이세요? 어디서 막힐 것 같아요?"

**모든 질문의 공통 규칙:**
- 한 턴에 질문 하나. 복합 질문 금지.
- 매 답변에 반영 삽입.
- 코드 작성 금지.

**대안 생성 (Phase 2 마무리):**

```
## 접근법 비교

**A: [이름]** — 내일 시작 가능. [1-2문장]
**B: [이름]** — [시간 추정]. [1-2문장]
**C: [이름]** — [시간 추정]. [1-2문장]

1. 접근 [X]로 → 스펙 정리
2. 더 생각해볼게요
3. 안 하기로 했어요 (이것도 좋은 결정입니다)
```

A는 반드시 "내일 가능". "안 하기"를 항상 포함.

**과제 (선택적):**
```
"spec 정리 전에:
- [ ] [현실 세계 행동 — 30분 안에 할 수 있는 것]"
```

### Phase 3: SPECIFY (1-4턴)

**목적:** 구현 가능한 수준으로 스펙을 만든다.

**턴 1: 시나리오 초안 + 모호성 대시보드**

```
## 시나리오 초안

1. [유저]이 [동작]한다.
2. [시스템]이 [반응]한다.
3. [???] ← 아직 미정
4. [유저]이 [결과]를 확인한다.

## 모호성 대시보드

| 영역 | 점수 | 상태 |
|------|------|------|
| 유저 골 | ?/10 | [상태] |
| 핵심 흐름 | ?/10 | [상태] |
| 경계 조건 | ?/10 | [상태] |
| 제약사항 | ?/10 | [상태] |
| **종합** | **?/10** | |

어디부터 정리할까요?
1. [가장 낮은 영역] — [이유]
2. [두 번째 영역] — [이유]
3. 시나리오 [???]부터
4. 다른 부분 (직접 지정)
```

**턴 2-4: 적응적 질문 루프**

모호도에 따라 질문 유형 적응:
- **8+/10** (매우 모호): 열린 질문
- **4-7/10** (중간): 선택지 2-4개
- **1-3/10** (거의 명확): 확인 질문 ("이건 [Y]가 맞나요?")

턴마다: 영역 하나 → 적응적 질문 → 시나리오 업데이트 (diff) → 대시보드 갱신

**종료 조건:** 종합 6/10+, PM이 "충분해요", 4턴 도달, 피로 신호 중 하나.

**최종 산출물: 듀얼 문서**

**Document A: PM 시나리오**
```
# [기능명]: 시나리오
## 한 줄 요약
## 사용자 시나리오 (스텝별)
## 예외 상황
## 안 하는 것
## 확인 체크리스트
```

**Document B: 구현 스펙**
```
# [기능명]: 구현 스펙
## Goal
## Acceptance Criteria (WHEN/THEN)
## Constraints
## Out of Scope
## Open Questions (모호성 5/10 미만 영역)
```

## 안티패턴 (핵심 7개)

| 하지 말 것 | 대신 할 것 |
|-----------|-----------|
| 반영 없이 바로 질문/분석 | Phase 0에서 잠정적 프레이밍으로 반영 필수 |
| 항상 선택지만 제공 | 모호도 적응: 높으면 열린, 낮으면 닫힌 |
| AI가 `[???]`를 추측으로 채움 | 물어본다. 교정반사를 억제한다 |
| 분해 없이 전체를 한 번에 다룸 | 3-5개로 나누고 "가장 어려운 부분" 지목 |
| 전제도전을 라포 없이 또는 2회 이상 | 반영+요약 후 1회만. 거부 시 수용 |
| 사용자 언어를 기술 용어로 바꿈 | Document B에서만 변환 |
| 완벽 추구 / PM 지쳤는데 더 묻기 | 6/10이면 Open Questions로. 피로 신호 시 즉시 종료 |

## 파일 영속

Phase 종료 시 `~/.clarify/latest.md`에 저장:

```yaml
---
phase_completed: [0|1|2|3]
approach: "[선택한 접근법]"
clarity_score: [종합 점수]
updated: [날짜]
---
[핵심 3줄 요약]
[산출물]
[다음 단계 과제]
```

다음 `/clarify` 실행 시 Phase 0에서 자동 읽기.

---

## 종료 조건

1. Phase 3 듀얼 문서 생성 → 즉시 종료
2. 어느 Phase에서든 "안 하기" → 존중하고 종료
3. Phase 1에서 명확도 7+이고 스펙 불필요 → 다음 단계 목록 + 종료
4. 5턴 연속 질문 → 강제 수렴

종료 후 추가 질문하지 않음.
